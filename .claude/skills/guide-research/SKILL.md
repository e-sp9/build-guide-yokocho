# キーボード設計者 作品調査・掲載選定

指定された設計者の全キーボード作品を網羅的に調査し、ビルドガイド横丁への掲載エントリを作成する。

引数: `$ARGUMENTS` — 設計者名（例: foostan）

## 基本方針

- Phase 5 でユーザーの明示承認を得るまでは、`src/content/guides/*.json` を作成・更新しない。
- 調査結果は、ユーザー承認前の提示表として「既存エントリ」「新規候補」「既存更新候補」「除外候補」「要承認」に分ける。
- 各判断には根拠 URL、確認日、または「未確認」を添える。検索結果の要約だけで確定扱いにしない。
- 設計者本人、ブランド、ショップ、共同設計、派生/再設計の帰属が曖昧な作品は即時追加せず「要承認」に分類する。`designer` 値を推測で決めない。
- ユーザー入力がブランド名やショップ名の場合も、ブランド製品を一律に同一設計者扱いしない。公式表記、原設計者、販売元の関係を提示表で分ける。

## Phase 1: 現状確認（並列実行）

以下を同時に実行する。

### 1a. 登録済みデータの確認
`src/content/guides/` 内のJSONファイルをすべて読み、指定設計者のエントリを一覧化する。

### 1b. JSONスキーマの確認
`src/content.config.ts` または既存JSONからフィールド定義を把握する。必須フィールド:
- `name`, `designer`, `description`(280文字以内), `url`, `tags`(配列), `addedAt`(YYYY-MM-DD)
- オプション: `descriptionEn`, `tagsEn`, `shopLinks`(配列 `{name, url}`)

## Phase 2: 網羅的調査（並列エージェント3本）

**必ず3本のエージェントを並列起動**し、異なるソースから情報を収集する。

Codex では `spawn_agent`、Claude Code では Task / Agent 相当の dispatch primitive を使う。エージェント起動が利用できない環境では、ユーザーにその制約を明示し、同じ3観点を逐次調査する。逐次調査に切り替えた場合は、最終報告の「裁量補完」にその旨を書く。

### Agent A: GitHub リポジトリ調査
- 設計者のGitHubプロフィールページから全リポジトリを取得
- 各リポジトリの README、docs/ ディレクトリを確認
- キーボード関連リポジトリとツール/ライブラリを分類

### Agent B: Web検索（日本語）
- 「{設計者名} 自作キーボード」「{設計者名} keyboard」で検索
- 既知の作品名があればそれでも検索
- BOOTH、ブログ、技術書の情報を収集

### Agent C: ショップ・コミュニティ調査
- 遊舎工房、TALP KEYBOARD 等の主要ショップで検索
- geekhack、Reddit の情報を確認
- 販売状況（現行品・終売品・GB済み）を把握

### 各エージェントが収集すべき情報
作品ごとに以下を報告させる:
- キーボード名
- 全バージョン/バリアント一覧
- GitHubリポジトリURL
- ビルドガイドURL（バージョンごと）
- 概要（キー数、分割/一体、スイッチ種別、特徴）
- 販売先と価格（判明している場合）
- 根拠 URL と確認日。購入可否は商品ページ上の購入導線、在庫表示、Sold out 表示などを根拠として書く

## Phase 3: ダブルチェック（並列エージェント）

Phase 2 の結果を突合し、**不一致・未確認の情報を検証する**。

### 検証対象
1. **エージェント間で片方しか報告していない作品** — 実在するか確認
2. **ビルドガイドURL** — WebFetch で存在・内容の充実度を確認
3. **README しかないリポジトリ** — キーボードプロジェクトか、ツール/ケース/テスト用かを判定
4. **バージョン一覧の網羅性** — docs/ ディレクトリ構造から全バージョンを確認

### 判定結果の分類

各作品を以下に分類する:

| 分類 | 条件 |
|------|------|
| **掲載対象** | ビルドガイド（組み立て手順書）が存在するキーボード |
| **掲載対象外: ガイドなし** | PCBデータのみ、ファームウェアのみ等でビルドガイドがないもの |
| **掲載対象外: 非キーボード** | ケース設計、ライブラリ、ファームウェアツール、書籍等 |
| **掲載対象外: 非ビルドガイド** | 設計チュートリアル、開発ガイド等（組み立て手順ではないもの） |

### 境界ケース

- あるキーボードが公式に別モデルのビルドガイドを参照している場合は、公式ドキュメントまたは README にその案内が確認できれば掲載対象にできる。表では「別モデルのガイドを流用」と明記する。
- 複数作品の build guide index リポジトリは単体では掲載しない。index 先にある個別キーボードを掲載候補に分解する。
- 設計者本人、ブランド、ショップ、共同設計の帰属が曖昧な作品は即時追加せず「要承認」に分類する。
- README しか確認できないキーボードは、組み立て手順が十分に含まれる場合のみ掲載候補にする。概要や写真だけなら「ガイドなし」に分類する。

## Phase 4: バージョン選定

掲載対象の各作品について、掲載するバージョンを決定する。

### 選定ルール

1. **単一バージョン** → そのまま掲載
2. **複数バージョンあり、最新版が主流** → 最新版のみ掲載
3. **最新版があるが旧版が有名/別物** → 両方掲載
   - 例: 標準版とLow-profile版は設計思想が異なる別製品 → 両方
   - 例: v3とv4で同じ路線の改良 → v4のみ

### バリアントの扱い
同じキーボードの**スイッチ違いバリアント**（Cherry/Chocolate/Light等）は**別製品として各々掲載**する。
各バリアント内のバージョン選定には上記ルールを適用する。

## Phase 5: エントリ作成

### ユーザーへの提示
実装前に以下を表形式で提示し、承認を得る:
- 掲載対象の全作品リスト（選定バージョン付き）
- 除外した作品とその理由
- 既存エントリ、新規候補、既存更新候補、要承認を分けた一覧
- 各作品の根拠 URL、確認日、shopLinks に入れる/入れない理由
- 帰属やバージョン選定が曖昧な作品では、ユーザーに承認してほしい判断点

承認前は JSON ファイルを作成・更新しない。ユーザーの明示的な「作成して」「更新して」「この内容で進めて」等の返答を承認とみなす。

### JSON作成ルール
- ファイル名: `{キーボード名のkebab-case}.json`（例: `corne-cherry.json`）
- バージョン違いで複数エントリがある場合: サフィックスで区別（例: `cornelius.json`, `cornelius-low-profile.json`）
- `description`: 製品の特徴を端的に。280文字以内。バージョン番号は含めない
- `url`: 日本語ビルドガイドを優先。なければ英語版
- `tags`: 既存タグとの一貫性を保つ。主要タグ:
  - 形状: `分割`, `一体型`, `マクロパッド`
  - サイズ: `40%`, `60%`, `75%` 等
  - 配列: `カラムスタッガード`, `オーソリニア`, `ロウスタッガード`
  - スイッチ: `MX互換`, `Choc互換`
  - 特徴: `ロープロファイル`, `ガスケットマウント`
- `shopLinks`: 現在購入可能なリンクのみ。終売品、在庫切れ、アクセサリ単体、PCB/ケース単体は省略
  - kit と prebuilt が両方購入可能な場合は、組み立て対象である kit を優先する
  - 複数ショップで同一 kit が購入可能な場合は、公式ショップまたは国内ユーザーに有用なショップを優先する
  - 「Select options」「Add to cart」等で購入導線が確認できるものは候補にできるが、在庫が不明な場合はユーザー提示表で注記する
- `addedAt`: 作成日（YYYY-MM-DD）

### 既存エントリ更新の扱い

既存エントリが見つかった作品は新規JSONを作らない。URL更新、description改善、tags調整、shopLinks追加が必要そうな場合は「既存更新候補」としてユーザー提示表に分ける。ユーザー承認なしに既存エントリを更新しない。

### ビルド確認
全エントリ作成後に `npm run build` を実行し、スキーマバリデーションとビルドが通ることを確認する。
